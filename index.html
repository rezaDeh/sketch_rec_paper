Abstract
In the mainly digitized world that we live in today, daily tasks have been computerized or have
the capability to be computerized. One of these tasks is electronic document signing. Many tools
such as DocuSign, and Adobe Sign have provided the mechanism for e-signatures. However, to our
knowledge currently there is no sketch level authentication system that identies if the intended
user has provided the e-signature themselves. The development of such system would be benecial
in many industries such backing, nance, and criminology. With the help of sketch recognition
technology and the widespread usage of touch screen tablets, we introduce AutheSign a sketch
based electronic signature authentication system. The main idea behind AutheSign is that it
contains a corpus of validated signature data for each of its users. It uses the features associated
with the veried user as a grand truth to authenticate any new signatures that are captured for
the said user. To test the performance of our system, we aim to conduct a small user study
with students from Texas A&M University and other selected individuals within our network to
determine how well AutheSign is able to dierentiate between real and forged signatures.
1 Introduction
There are many authentication tools used to validate user identities. However, when it comes
to electronic signatures, there exists no such thing as an e-signature authenticator. Whether
it is signing a simple banking document or a multi-million dollar business deal, an added level
of protection in terms of signature verication would produce a major obstacle for fraudulent
activities. This is because even if the individual contains all the security related information of
an authenticated user, it will be extremely dicult for them to accurately forge their signature.
With AutheSign we aim to use a small subset of the Rubine Features [8]; specically, Squared
Sum of angles traversed (RF11) and total duration of gesture(RF13) as well as Long Feature 16
[5] which is the stroke length divided by the bounding box diagonal length. Additionally, we
will use pressure data as well as the starting position within the sketch. Having captured these
features from each user, we will use an augmented template matching algorithm to compare the
new signature data with the corpus of authenticated user signature data previously captured.

3 Methodology
The aim of this work is to identify (1) if the mentioned features are sucient enough to accu-
rately dierentiate between authentic and forged signatures? (2) would an augmented template
matching algorithm capable of handling multi-sketch data suce for this task? (3) How large of

Is it possible to rank the features in terms of authentication importance?
3.1 Data Gathering
Two sketch accessible platforms for gathering sketch data were designed for this task. The rst
one, was a website build on Flask and hosted on Heroku and the second one was accessible through
a native SwiftUI app. Both platforms connected via endpoints to the the Firebase database.
3.1.1 Flask Website
The Flask website was our rst attempt at creating a stable and secure platform for gathering
sketch data. The website was guarded by a Google login, and all submissions needed to go through
an authorized account. From there, the data was collected using a standard JavaScript script which
took the inputs produced by the stylus and stored them in an array of strokes. Every stroke was
an array of data points which contained x, y, pressure, time, and two angle metrics. Each users
name was encrypted using JWT and real signatures and copies of signatures (frauds) had their
own spaces. The original database design concatenated the data points into a large string, which
resulted in issues with string size limits. The updated database design keeps every signature as
its own string, and then the authentic signatures would be a list of strings in the database. To
prevent any issues, the strings were compressed to save more space as well.
3.1.2 SwiftUI App
The SwiftUI app is a native iPadOS app that was created as a result of the websites inability to
accurately detect whether a sketch point is caused by a stylus or another object. To neutralize
this issue, we decided to create a native iPadOS app to enable the gathering of better quality data.
An instance of noise data collection would be when, clicking the submit button on the website
leads to points being gathered as a part of the sketch. SwiftUIs utility based libraries allow for
a wide range functionalities on touch accessible apps. This This Change from a web-based UI
to a App based UI improved data quality signicantly. Furthermore, the native app used similar
endpoints as the ask app, but did get its own set of Endpoints, as we did not want to go through
the process of creating a Google login on an app which was restricted to only the people who were
on TestFlight. The app gathered the same data shape and information as the website did, thus
preventing any further modication of existing code to handle the new gathering method.
3.2 Data Pre-Processing
When it comes to leveraging data to either create a classication model or generate important
insights needed for decision making, it is important to transform raw data into an understandable
and readable format, or generally enhance the quality of the data, so as to promote the extraction
3
of meaningful insights. The data collected for this project came in a JSON format, and contained
x, y, time, and pressure data, as well as altitude and azimuth angles.
Before performing any preprocessing of the data, it was necessary to rst look through the data
to determine the kind of preprocessing that would be needed. We did therefore perform a data
parsing from deeply nested JSON objects into a more readable format, showing the signatures
signed by each user and the amount of strokes each signature is made up of. Upon parsing the
data, we found that there is an inconsistency in the number of strokes that each user used to sign
their signatures - across multiple trials. This observation therefore necessitated a concatenation
of multi-stroke signatures, so that each signature consists entirely of a single stroke.
With multi-stroke signatures combined into single-stroke equivalents through concatenation,
a straight line automatically gets drawn that connects the last point in each stroke with the
rst point in the next stroke (if present). While stroke concatenation was necessary to ensure
consistency in the number of strokes per signature of each user, it did also help to spot some
artifacts in the stroke collections (gure 1). These are tiny strokes, each containing anywhere
between 2 and 4 points, that users unintentionally created by making stylus pen-screen contacts
before and/or after their signatures. Some artifacts also came from ngertip contacts with the
screen during some signature collection attempts. Another important property of these artifacts,
as seen in gure *, is that they are located very far away (on the screen) from the other strokes,
and contain points with x, y and time values that are oddly too large or smaller than the x, y and
time values seen in other stroke collections (gure 2). These outliers were therefore removed to
enhance the quality of the data used to generate our heuristics.

3.3 Feature Extraction
Many vision based classication systems use featureless and raw data methods for dierentiating
between shapes. However for the sake of simplicity and considering the wide range of shapes that
we will encounter, we decided as a nal pre-processing step to perform feature extraction on the
data to generate useful features upon which our heuristics is based on. This process took the
x, y, time, pressure, altitude and azimuth angle values recorded for each signature on the sketch
interface, and calculates a total of 41 gesture-based features based on them. The extracted features
include all Rubine and Long features, as well as their derivatives.
3.4 Authenticator
3.4.1 Augmented Template Matching
This was the original approach considered for this task. Its driver was a template matching
system where inputs are compared to examples of each class to classify them. Having a corpus of
authenticated signatures from each user, it is natural to consider these as our template. However,
5
since we care about how the stroke is drawn we need to consider the extracted feature data in
our template instead of considering how the sketch looks. Additionally we need to apply an
aggregation to consider features from all the strokes within a single signature.
3.4.2 Heuristic Based
After much deliberation, we decided that creating and training a classier for every user was
not only slow, but also impractical and unfeasible. Thus, we set out to create a heuristic which
could accurately determine whether a signature was real or not, by only needing at least two real
signatures. From here, we gured that template matching was a great place to start from and
work backwards. Ultimately we recognized that it is not necessarily template matching, but also
does t the description in its own right. The way the heuristic works is quite simple. Every new
(authentic) signature which gets submitted has its features generated, as described in the previous
section. From here, these features are also uploaded to the users database. If the user has at
least two signatures in the database, then they can start having their signatures classied. This
is done by taking the mean and standard deviation of each of the features of the users authentic
signatures. We then calculate the feature values of the signatures we are trying to test. For every
feature of the signatures we are checking, we check if it falls within one standard deviation of the
mean from the previously calculated authentic signatures. If it does, that feature gets a Yes vote,
or 1. If it falls outside that threshold (mean 1 std) then this feature gets a No vote, or 0. After
we do this for every feature, we have a list of 1s and 0s. From here we can take the mean and
given a threshold we can decide if the votes say that the signature we are testing is an authentic
signature or a fraud. We currently have this threshold set to 0.5, but a threshold analysis can
easily be done in a later stage to optimize the performance. This could be done, but for example
calculating the F1 score of every threshold between 0.01 and 0.99 and then choose the threshold
with the largest F1 score.
4 Results
In order to evaluate AutheSign The original plan was to conduct a controlled study. we aimed to
conduct an ongoing study on N dierent participants. Each individual was to produce X dierent
sketches of their own signature and X-1 sketches replicating the other participants sketches. For
the replication task we will allow each user to practice replicating each X-1 signature at most M
times before producing the nal forged signature. Given the scope of this project and the time
allotted, out user study did not go as planned and the there was not a controlled number real and
forged signatures as well as no focus on ensuring signature quality for each user sketch.
Using this heuristic on all the data we gathered, and without performing any threshold op-
timization i.e. keeping the threshold at 0.5, we were able to achieve an accuracy of 0.88. The
heuristic was able to classify all 133 fraudulent signatures as frauds. Of the 190 real signatures, it
was able to classify 152 correctly and classied 38 as fraudulent. This leaves us with an F1 score
of 0.89.

5 Discussion
As we witnessed the performance of the heuristic approach struggles a bit with classifying real
signatures as real. There are several factors which play into this, for one, we think that optimizing
the decision threshold could prove critical for improving the accuracy of this heuristic to be over
0.9. Further, there is something to be said about the signatures of each of the users. For one,
having more signatures stabilizes the heuristic in that the mean and standard deviation wont
move a lot for every new authentic signatures, which are added. Further, the variability of a users
signature plays a large role in the ability to accurately classify a signature. If a user has a large
band of possible signatures, then the heuristic will perform markedly worse than if the signature
is very consistent, as the mean and standard deviation of the features will be tighter. However,
given that this heuristic works with only submitting two authentic signatures, we believe that the
heuristic already performs quite well.
